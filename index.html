Zero-Trust AI Workshop

Funny landing page. Serious blast radius.

The point of the workshop is simple: coding agents are useful, but clicking accept on a command you do not understand is how people end up deleting files, leaking secrets, or proudly breaking their own machine. So the workshop now ships with a portable shield, not just a warning.

What changed

From “please be careful” to “the system refuses stupid.”

This workshop now frames safe AI coding as a product pattern people can deploy: a secure lab box for execution, a rules engine that blocks risky actions, and a self-hardening harness that learns from failed tests without ever removing the walls.

Layer 1

Lab Box

Run agent actions in a contained workspace with no ambient host trust, no random secrets, and no free outbound network.

Layer 2

Policy Shield

Inspect every shell, file, and network action against explicit rules for commands, paths, domains, and write boundaries.

Layer 3

Harness Loop

Feed blocked actions back into tests so the wrappers improve, while the hard policy remains the authority.

Policy

policy.yaml

Explicit allow and block rules for commands, paths, and outbound access.

allowed_commands:
  - pytest
  - git status
  - rg
blocked_patterns:
  - "rm -rf"
  - "curl .*@.env"
network:
  enabled: false

Skill

agent-shield.skill.md

A drop-in operating contract that tells the model how to behave under zero-trust conditions.

You are operating behind a policy shield.
Never assume shell access is safe.
Explain intent before proposing commands.
Treat blocked actions as feedback, not permission to retry elsewhere.

Workshop Flow

A structure non-technical users can follow

Keep the story plain: the AI works inside a secure lab box, the shield decides what is allowed, and the harness improves wrappers by learning from blocked mistakes.

00:00 - 00:15

The disaster tour

Show how fast a copied command can delete work, leak a secret, or rewrite shared history. Make the risk legible before talking about mitigation.

00:15 - 00:35

The secure lab box

Explain sandboxing, bounded filesystems, no ambient credentials, and why agents should never have direct trust on user machines.

00:35 - 00:55

Policy in plain English

Walk through allowed commands, blocked paths, and why “the agent asked nicely” is not a security control.

00:55 - 01:20

Harness the harness

Demonstrate how failed or blocked agent actions become tests that refine safe wrappers without widening access.

01:20 - 01:30

Takeaway pack

Send people away with the shield files, a short operating checklist, and one rule: no command approval without comprehension.

Danger Zone

Commands that make a great live demo and a terrible habit.

These examples are useful because non-technical users immediately understand the point: if an agent can suggest them, you need a layer that blocks them before curiosity turns into cleanup.

File deletion

rm -rf

Deletes recursively and fast. Perfect example of why approval prompts are not enough.

Disk overwrite

dd if=... of=/dev/diskX

Raw disk writes are not a learning exercise. One wrong target and you are rebuilding a machine.

Filesystem format

mkfs / diskutil eraseDisk

Great reminder that agents should not have access to destructive admin paths by default.

Git rewrite

git reset --hard / git push --force

These are recoverable only until they are not. Good workshops show the blast radius before the shortcut.

Secret exposure

cat .env / printenv

Secrets do not become safe because they were revealed by a helpful assistant.

Phone home

curl / wget

Outbound network is where “show me the logs” quietly turns into data exfiltration.
